弱内存模型的 CPU 乱序问题检测方法

有些 cpu 为了提高性能，引入了一些优化手段，导致指令实际执行的 “效果” 可能与程序顺序不同，具有这种内存模型的 cpu 通常被称为 ”弱内存模型“。（可以参考 ibm 的一篇文章，看一下 store buffer 和 invalid queue 怎么影响指令执行顺序），同时，还有另外一些原因可能导致指令乱序的效果：

  • 编译器对指令做重排序
  • cache 一致性优化引入的乱序
  • 指令多发射，有的流水线的指令由于没有阻塞而先执行，有的被阻塞了后执行从而导致顺序变化

指令乱序的最小约束条件

看起来比较反直觉，毕竟大多数同学写程序时都不会考虑乱序问题，为什么从不出错？

有几点原因，一个是对单线程来说，即便乱序发生了也不会影响程序的执行结果，另一个是在多线程程序中，对临界区加锁和解锁本身就自带了内存屏障语义，所以也不会出错。一般在多线程下写无锁算法时才需要考虑乱序，这个一会分析。

所以首先回答一个关键问题，乱序的边界在哪里？是所有指令都能乱序，还是有一定的约束？

事实上，为了保证程序的正确性，不管是编译器的指令重排序还是 cpu 指令执行乱序或者是 cache 一致性优化引入的乱序，指令的可见性顺序必须满足单线程语义下的正确性。换句话说，在指令执行的每个时间点上对内存的可见性顺序必须和程序顺序保持一致。

这里用可见性顺序而不是执行顺序去描述，是因为指令的执行过程实际分很多步骤，包括取指、译码、执行、访存、写回等，可能存在有指令取指和译码先进行但等到执行和访存时需要等待其它指令结束的情况，用执行顺序分析会引入太多不必要的概念，徒增分析的复杂度，而内存可见性顺序则抽象了这个问题中最本质的需求。

多线程下的乱序问题

根据上述讨论可以知道，单线程情况下即便有乱序行为发生也不会影响程序的执行结果，所以无需担心。然而，在多线程下仅凭以上的约束就不足以保证程序的正确性了。比如以下程序：

所以，多线程下乱序会引起问题，本质上是因为不管是编译器还是 cpu，在处理指令时其本身并没有 “线程” 的概念，无法从多线程角度对指令的执行增加新的约束（cpu 做不到我认为是代价太大，也没必要。编译器有一些方案会考虑并自动加内存屏障）。

解决乱序问题

既然编译器和 cpu 都没有线程概念，那需要的 ”约束“ 条件就需要程序员自己来加了，这就引入了 ”内存屏障“ （memory barrier）。对上面的程序来说，需要保证两件事：

  1. 从 T1 的角度来看，当 b = true 对外部可见时，a = 123 也必须对外部可见，这个约束叫做 release 语义（通常实现为 write barrier）
  2. 从 T2 的角度来看，当 b = true 可见时，如果 T1 的 b = true 具有 release 语义，那么 happens before b = true 的所有指令的结果对 T2 也必须可见，这种叫做 acquire 语义（通常实现为 read barrier）

a = 123 和 print(a)，前者写 a，后者读 a，并且在两个线程中没有加锁，熟悉多线程的就会发现这就是一个 data race，当然如果读写 a 都是原子的并没有什么问题，不过这并不影响这两个操作是并发关系，即两个操作相互没有 happens before 关系。

那加了内存屏障后呢，当 b = true 被 T2 可见时，意味着 Wb@T1 happens before Rb@T2，又由于内存屏障的存在，保证了 T2 此时读到的 a 必然是 123，从结果上推得 Wa@T1 happens before Ra@T2。看起来用 happens before 关系就能描述可见性顺序。

试着用 happens before 关系去描述可见性顺序，在 b = true 对 T2 可见时：

  1. 如果 Wa@T1 与 Ra@T2 之间是并发关系，那么 Wa@T1 是否被 T2 可见不能保证
  2. 如果 Wa@T1 happens before Ra@T2，那么 Wa@T1 能被 T2 可见

那么寻找哪些变量的可见性顺序可能变化可以转为寻找变量的写-读是否存在并发，也就是 wr-race。

另外可以观察到一点，T1 的写顺序和 T2 的读顺序往往是相反的，这也很容易理解，因为既然 T2 的两个读操作之间是有依赖关系的，那么假设这两个读操作的顺序和 T1 写操作顺序一致，即便没有乱序发生，也不能保证第二个读看到的内存值是什么，这样的程序本身就是不合理的。

因此，解决几点问题可以在尽量小的误报率下检测出乱序问题：

  1. 寻找 wr-race
  2. 每个线程中的内存访问操作标记本地逻辑时间 epoch，当寻找到某个 wr-raceA 后，查询是否存在 wr-raceB 满足：wr-raceA[W].epoch < wr-raceB[W].epoch && wr-raceA[R].epoch > wr-raceB[R].epoch
  3. 判断读线程的两个读操作之间有没有依赖关系
